Phase 03 · stages 3.1 – 3.5

Template Understanding

Parse the blank target template to understand exactly what the user wants generated.

How it works

The AI acts as a "Blueprint Scanner." Before it can write anything, it reads your empty template, figures out the required chapters, finds all the missing data it needs to fill in, and maps out the exact order in which it needs to write.

3.1

Parse Template

Reads the blank Markdown template and turns it into a structured hierarchy the AI can understand.

What this stage is doing

Takes the raw Markdown file uploaded by the user and parses its headers and syntax, converting it into a clean, hierarchical document tree.

Why it is useful

By strictly enforcing a Markdown-only pipeline, it ensures the AI can quickly and accurately navigate the template without complex parsing errors.

Problem Solved

The Formatting Inconsistency Problem: Markdown provides a rigid, predictable structure, preventing the AI from getting confused by the hidden formatting bugs found in rich-text documents.

Data State PARSED 
{
  "document": "SRS_Template_v2.md",
  "type": "markdown_hierarchy",
  "root_nodes": 12,
  "status": "ready_for_scan"
}
3.2

Identify Sections

Scans the blueprint to find all the chapters, headings, and appendices.

What this stage is doing

Reads through the document tree to locate every structural boundary and builds a complete Table of Contents.

Why it is useful

Sets the exact skeleton of the final document so the writing agents know exactly what chapters they need to write.

Problem Solved

The Scope Creep Problem: By locking down the required chapters early, it prevents the AI from hallucinating extra sections or skipping important ones.

Data State STRUCTURED 
# Table of Contents Detected
1.0 Introduction
2.0 Hardware Architecture
  2.1 Power Systems
  2.2 Interfaces
3.0 Software Requirements
3.3

Identify Required Fields

Finds all the empty placeholders (like {{VOLTAGE}}) and blank tables that need to be filled.

What this stage is doing

Acts as a targeted search engine, finding every single blank space, tag, or "TBD" marker inside the template.

Why it is useful

Creates a strict "Shopping List" of the exact data points the AI needs to fetch from the Memory Store later on.

Problem Solved

The Missing Data Problem: Catalogs every blank space so the system can automatically check if the document is actually complete before finalizing it.

Data State EXTRACTED 
// Missing Variables Found
- {{PROJECT_NAME}} (Header)
- {{MAX_VOLTAGE_RATING}} (Sec 2.1)
- [Insert Interface Table] (Sec 2.2)
3.4

Build Dependency Graph

Figures out which sections rely on each other and must be written first.

What this stage is doing

Maps out connections. If the "Thermal Limits" chapter requires the "Power" chapter to be written first, it creates a workflow map enforcing that order.

Why it is useful

Dictates the exact chronological order in which the AI agents must write the document.

Problem Solved

The Context Error Problem: Prevents the AI from trying to summarize the power usage before it even knows what the power specs are.

Data State MAPPED 
# Generation Order
1. Section 2.1 (Power Systems)
2. Section 2.2 (Interfaces)
3. Section 4.0 (Thermal - DEPENDS ON 2.1)
3.5

Determine Output Types

Checks if a section requires a data table, a bulleted list, or narrative paragraphs.

What this stage is doing

Looks at the placeholders and figures out the required visual format for the injected data.

Why it is useful

Tells the final writing agents exactly how to format the data when they inject it into the template.

Problem Solved

The Formatting Disaster Problem: Stops the AI from taking 50 hardware pins and dumping them as a giant unreadable paragraph instead of the neat table the template requires.

Data State STYLED
{ "section": "2.2 Interfaces", "required_format": "TABLE", "columns": ["Name", "Speed", "Interface Description"] }
