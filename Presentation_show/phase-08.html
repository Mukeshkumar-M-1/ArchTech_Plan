Phase 08 · stages 8.1 – 8.4

Template Memory Update

Write the user's answers back into the knowledge base — then re-check everything.

678 part of the verification loop · repeats until no gaps remain
How it works

After the user provides clarifications in Phase 7, this agent integrates those answers deeply into the system's memory banks, ensuring the data is permanently saved and ready for the gap re-verification loop.

8.1

Update Existing Memories

Modifies existing markdown sheets with the new values.

What this stage is doing

Takes the structured user answers and safely injects them into the specific existing memory markdown files using idempotent writes.

Why it is useful

Enriches the knowledge base without creating messy duplicate files.

Problem Solved

The Outdated Knowledge Problem: Keeps the memory store dynamically updated as the project evolves.

Data State UPDATED 
{
  "file": "memory/hardware_specs.md",
  "action": "Idempotent Write",
  "key_updated": "clock_speed_hz",
  "status": "SUCCESS"
}
8.2

Create New Memory Files

Drafts entirely new memory files if the user introduced a new topic.

What this stage is doing

If a user's answer describes an entirely new subsystem, this agent generates a brand new memory file from scratch.

Why it is useful

Handles scope creep and expands the architecture plan dynamically.

Problem Solved

The Missing File Problem: Prevents data from being dropped just because a folder didn't previously exist.

Data State CREATED 
{
  "new_file": "memory/bluetooth_module.md",
  "category": "Hardware/Wireless",
  "trigger": "User Clarification",
  "status": "INITIALIZED"
}
8.3

Update memory.md Index

Refreshes the global directory table of contents.

What this stage is doing

Scans the entire memory store directory and updates the MEMORY.md master index file with links to any new or renamed files.

Why it is useful

Acts as a directory map so other agents can easily navigate the file system.

Problem Solved

The Dead Link Problem: Ensures that all cross-references in the system remain valid.

Data State INDEXED 
{
  "file_scanned": 15,
  "new_entries_added": 1,
  "index_file": "MEMORY.md",
  "status": "SYNCED"
}
8.4

Rebuild Memory Files

Rebuilds affected memory files cleanly with the user's inputs merged in.

What this stage is doing

Reconstructs the full Markdown structure using the updated JSON data, ensuring that the final file is perfectly formatted.

Why it is useful

Maintains strict adherence to the Markdown and JSON-only architecture, avoiding complex databases.

Problem Solved

The Format Drift Problem: Ensures the memory files stay pristine and human-readable after automated updates.

Data State REBUILT 
{
  "file": "memory/hardware_specs.md",
  "format": "MARKDOWN",
  "status": "CLEAN_REBUILD",
  "action": "READY_FOR_VERIFICATION"
}

Once memories are rebuilt, the pipeline loops back to Phase 6 to re-run gap detection — confirming the update actually closed the gap rather than assuming it did.
