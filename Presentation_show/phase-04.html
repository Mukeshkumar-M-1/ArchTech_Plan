Phase 04 · stages 4.1 – 4.5

Template Architecture Planning

Design a blueprint detailing how the document will actually be built.

How it works

Once Phase 3 has scanned the empty template, this phase compares that empty blueprint against all the knowledge in the Memory Store (from Phase 2). It decides exactly how the knowledge will fit into the template before any actual writing begins.

4.1

Analyze Template Structure

Compares the template's broad layout against available memory categories.

What this stage is doing

Takes the Document Tree from Phase 3 and checks if the required knowledge actually exists in the Memory Store (e.g., if the template has a "Software" section, it checks if Software memories exist).

Why it is useful

Performs a high-level feasibility check before wasting time and compute resources on detailed mapping.

Problem Solved

The Total Mismatch Problem: If a user uploads a Mechanical template but the memory store only has Software data, this stage flags the massive architectural failure immediately.

Data State CHECKED 
{
  "template_domain": "Hardware Specs",
  "memory_domains": ["Hardware/Power", "Hardware/Clocks"],
  "match_status": "PASS"
}
4.2

Analyze Template Sections

Determines which engineering domain belongs in which specific chapter.

What this stage is doing

Profiles each chapter in the template to lock down its exact engineering topic (e.g., locking "Section 3.1: Power Rails" strictly to the `Hardware/Power` memory category).

Why it is useful

Assigns a specific "topic scope" to each chapter, preventing AI writing agents from rambling off-topic.

Problem Solved

The Topic Bleed Problem: Prevents software specifications from accidentally being injected into a chapter that is supposed to be strictly about hardware pinouts.

Data State PROFILED 
# Chapter Constraints
Chapter: 2.1 Power Systems
Allowed Memories: categories/power.md
Restricted: Software, Mechanics
4.3

Analyze Template Required Fields

Maps specific memory variables to template placeholders.

What this stage is doing

Takes the empty placeholders from Phase 3 (like `{{CLOCK_SPEED}}`) and searches the Memory Store to find the exact file that contains that specific value.

Why it is useful

Creates a hard, verified link between the empty template slot and the exact data source.

Problem Solved

The Hallucination Problem: By strictly mapping placeholders to real data before generation, it prevents the AI from inventing numbers when it actually writes the text.

Data State MAPPED 
{
  "placeholder": "{{CLOCK_SPEED}}",
  "source_file": "requirements/REQ-042.md",
  "json_key": "max_freq_mhz"
}
4.4

Analyze Template Dependency Graph

Creates the step-by-step queue of which sections must be generated first.

What this stage is doing

Turns the dependency graph from Phase 3 into an actual Execution Schedule. It schedules independent sections to be written first, and summary sections to be written last.

Why it is useful

Optimizes the generation process, allowing some chapters to be written in parallel while safely holding back dependent chapters.

Problem Solved

The Race Condition Problem: Prevents the system from trying to write the "Executive Summary" before the individual technical chapters have even been drafted.

Data State SCHEDULED 
# Execution Queue
Batch 1: [Sec 2.1, Sec 2.2, Sec 2.3]
Batch 2: [Sec 3.0 (Depends on Batch 1)]
Batch 3: [Sec 1.0 (Executive Summary)]
4.5

Analyze Template Output Types

Plans how data will be converted to match the requested format (e.g., text to table).

What this stage is doing

Looks at the formatting rules from Phase 3 and formulates a conversion strategy. If the template needs a table, but the memory store has a paragraph, it plans a conversion step.

Why it is useful

Prepares the necessary formatting tools (like markdown table generators) ahead of time.

Problem Solved

The Format Clash Problem: Prevents the system from blindly pasting raw paragraphs into a rigid column layout, which would break the final document.

Data State PLANNED 
{
  "target": "Interface Section",
  "source_data": "Paragraph descriptions",
  "required_format": "Markdown Table",
  "conversion_tool": "Table_Generator_Agent"
}
