Phase 09 · stages 9.1 – 9.5

Template Final Assembly

Compile every mapped, gap-free section into one integrated draft.

How it works

Once all content is mapped and the Gap Detection gate confirms everything is 100% complete, this phase takes over. It acts as the final "Book Binder," stitching all individual pieces into a single, cohesive document.

9.1

Merge Sections

Concatenates all individual memory files in the correct order.

What this stage is doing

Takes the ordered list of files produced by the execution schedule and physically appends them sequentially into a single target file using low-memory streaming.

Why it is useful

Creates the final book from the chapters without causing a memory overflow on the server.

Problem Solved

The Disjointed File Problem: Replaces hundreds of tiny memory files with one easily readable specification sheet.

Data State STREAMING 
{
  "action": "File Append (>>)",
  "source": "memory/ch03_architecture.md",
  "target": "build/final_draft.md",
  "status": "MERGING_SECTION_4_OF_20"
}
9.2

Build TOC

Scans the merged draft and automatically generates a Table of Contents.

What this stage is doing

Scans the merged buffer for Markdown headers (e.g., #, ##) and compiles a hierarchical TOC at the top of the document.

Why it is useful

Makes the massive document immediately navigable.

Problem Solved

The Navigation Problem: Prevents the user from having to scroll blindly through 100 pages of text.

Data State COMPILED 
## Table of Contents
- [1. Executive Summary](#1-executive-summary)
- [2. System Architecture](#2-system-architecture)
  - [2.1 Hardware Specs](#21-hardware-specs)
9.3

Apply Formatting

Enforces consistent markdown styling across the entire book.

What this stage is doing

Runs a formatting script over the text to ensure uniformity (e.g., all tables have standard markdown pipes, headers have correct spacing).

Why it is useful

Gives the document a highly professional visual polish.

Problem Solved

The Visual Clutter Problem: Fixes ugly text wrapping or broken tables caused by different LLMs generating slightly different formats.

Data State FORMATTED 
> run markdown-linter final_draft.md --fix
[OK] Normalized 14 tables
[OK] Fixed 3 header spacings
[OK] Removed trailing whitespaces
9.4

Resolve References

Fixes internal links so clicking a reference jumps to the right section.

What this stage is doing

Finds internal reference tags like [See Section 4.2] and updates them so they correctly link to the generated anchor tags.

Why it is useful

Makes the final Markdown document fully interactive and traceable.

Problem Solved

The Dead Link Problem: Ensures cross-references aren't broken when sections are re-numbered during merging.

Data State LINKED 
{
  "reference_tag": "[See Section 4.2]",
  "resolved_anchor": "#42-thermal-constraints",
  "status": "LINK_VERIFIED"
}
9.5

Generate Final Document

Outputs the absolute final version of the unified text.

What this stage is doing

Finalizes the text buffer and saves it to the disk as the golden master copy.

Why it is useful

This is the final phase before the document leaves the internal system pipeline.

Problem Solved

The Finalization Problem: Locks the draft so no further autonomous agents can modify the text.
