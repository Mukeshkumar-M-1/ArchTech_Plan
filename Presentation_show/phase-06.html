Phase 06 · stages 6.1 – 6.4

Template Gap Detection

Identify any missing data or broken dependencies before a single word of final text is generated.

678 part of the verification loop · repeats until no gaps remain
How it works

This is the ultimate "Quality Assurance" gate in the pipeline. Before generating any actual text, this agent scans the mapped template to ensure every single required detail is present, correctly formatted, and structurally sound.

6.1

Detect Missing Fields

Scans the template for empty placeholders.

What this stage is doing

Performs a rigorous check of all placeholders. If a placeholder was mapped in Phase 5 but the memory store had no data for it, this stage flags it.

Why it is useful

Prevents the final document from containing embarrassing blanks like "The required voltage is [INSERT VOLTAGE HERE]."

Problem Solved

The Null Data Problem: Ensures absolute data completeness before proceeding.

Data State FLAGGED 
{
  "field": "{{MAX_VOLTAGE}}",
  "status": "NULL",
  "error": "Data not found in Memory Store",
  "action": "Route to Phase 7"
}
6.2

Detect Incomplete Content

Checks if the mapped memory provides enough detail for the section.

What this stage is doing

Evaluates the quality and length of the mapped content. If a "Detailed Architecture" section only received a single sentence, it gets flagged.

Why it is useful

Guarantees that the final document meets professional depth and length standards.

Problem Solved

The Superficial Content Problem: Prevents the generation of bare-bones or heavily unbalanced chapters.

Data State FLAGGED 
{
  "section": "3.1 Architecture Overview",
  "status": "INCOMPLETE (Word count: 14)",
  "loop_count": 2,
  "action": "OFFERING_SKIP_BYPASS"
}
6.3

Detect Missing Dependencies

Verifies that all upstream variables required by a section are available.

What this stage is doing

Re-validates the execution queue. Ensures that if Section B relies on a parameter from Section A, that parameter actually exists.

Why it is useful

Prevents cascading failures where one missing piece of data ruins multiple downstream chapters.

Problem Solved

The Broken Chain Problem: Catches architectural logic flaws before they corrupt the final document.

Data State FLAGGED 
{
  "target": "Chapter 4 (Thermal)",
  "missing_dependency": "Power_Output_Max",
  "status": "CHAIN_BROKEN"
}
6.4

Detect Missing Output Types

Checks if formatting rules are missing for specific sections.

What this stage is doing

Ensures every section has an assigned, validated style template.

Why it is useful

Prevents formatting engine crashes during the final assembly.

Problem Solved

The Unformatted Text Problem: Ensures 100% styling coverage for the entire document.

Data State FLAGGED 
{
  "section": "Appendix A",
  "target_format": "UNKNOWN",
  "status": "STYLE_ERROR"
}
If gaps detected: yes Routes forward to Phase 7 — the user is asked to clarify the missing pieces.
If gaps detected: no Skips ahead to Phase 9 — nothing is missing, so assembly can begin.
