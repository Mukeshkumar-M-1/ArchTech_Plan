Phase 01 · stages 1.1 – 1.8

Data Extraction Pipeline

Convert messy, complicated PDFs into clean, easy-to-read instructions.

How it works

The system reads the document page-by-page, cleans up the messy text, and finds the actual rules. It then rewrites those rules to sound professional, sorts them into folders, explains what they mean, and removes any duplicates.

1.1

Ingest

Reads your uploaded documents and pulls out all the text, tables, and important rules.

What this stage is doing

  • Scans the uploaded document page by page, pulling out text and tables.
  • If a page is an image or scan (like a photo of a document), it uses OCR to read the text anyway.

Why it is useful

It makes sure the system can read any type of file, whether it's a typed digital document or a scanned piece of paper.

Problem Solved

The Unreadable Document Problem: Scanned pages usually break computer programs. This step guarantees we capture every single word, no matter how the document was made.

Data State RAW 
 Page 14 ... \n The sys \n tem should support spi approx 10MHz ...~~$
1.2

Segment

Breaks the text into neat paragraphs and throws away useless things like page numbers and headers.

What this stage is doing

  • Organizes messy text into clear paragraphs.
  • Connects broken sentences back together so they make sense.
  • Splits giant blocks of text into smaller, readable chunks.
  • Automatically skips cover pages and the table of contents.

Why it is useful

It clears away the junk, so the AI only focuses on the actual important rules, saving time and preventing confusion.

Problem Solved

The Cluttered Text Problem: Standard PDF readers often mix page numbers and footers into the middle of sentences. This step fixes all of that mess.

Data State SEGMENTED 
The system should support spi approx 10MHz.
1.3

Detect

Uses intelligent scoring to isolate the actual engineering requirements from general background descriptions.

What this stage is doing

It gives a score (from 0 to 10) to every sentence to see if it's an important rule:

  • High Score: Contains strict words like "must" or "shall".
  • Medium Score: Contains measurements (like volts, speed, or temperature).
  • Low Score: Vague words like "maybe" or "approximately".

Why it is useful

It separates the strict instructions that need to be built from the general descriptions that are just there for context.

Problem Solved

The Storytelling Problem: Documents often have long, rambling introductions. This step throws away the fluff and keeps only the real requirements.

Data State DETECTED 
Score: 8/10 [Pass]
Reason: Directives found ("should"), Protocol ("spi"), Unit ("MHz")
Text: The system should support spi approx 10MHz.
1.4

Normalize

Rewrites every rule so they all use the exact same professional tone and format.

What this stage is doing

  • Changes weak suggestions (like "will", "should", or "needs to") into the strict, professional word: "shall".
  • Removes confusing or vague phrases like "if necessary" or "as needed".
  • Fixes grammar, capitalization, and double-spacing so everything looks uniform.

Why it is useful

It makes the entire document extremely easy to read, because every instruction follows the exact same pattern.

Problem Solved

The Multiple Authors Problem: When different people write a document, the tone is messy. This step makes it look like one professional author wrote the whole thing.

Data State NORMALIZED 
The system shall support SPI, 10MHz.
1.5

Classify

Automatically sorts every rule into specific folders like Power, Software, or Safety.

What this stage is doing

It reads each rule and tags it into a specific department, such as:

  • Hardware: Power, memory, screens, and physical parts.
  • Software: Code, apps, and digital drivers.
  • General: Safety, security, and performance speed.

Why it is useful

It groups similar tasks together, so the software team only sees software rules, and the hardware team only sees hardware rules.

Problem Solved

The Manual Sorting Problem: Humans used to spend days reading thousands of lines to group them into folders. The AI now does this instantly.

Data State CLASSIFIED 
{
  "domain": "Hardware/Interface",
  "text": "The system shall support SPI, 10MHz."
}
1.6

Explain

Adds a clear, simple explanation for exactly what the rule means and why it exists.

What this stage is doing

  • Meaning: Translates complex technical jargon into plain English.
  • Components: Lists exactly which parts or systems are involved.
  • Purpose: Explains the real-world reason why this rule is necessary.

Why it is useful

It helps developers understand the big picture, preventing mistakes caused by blindly following instructions without knowing why.

Problem Solved

The "Magic Numbers" Problem: Often, a rule will just list a random number or setting with no explanation. This stage adds the missing context so anyone can understand it.

Data State EXPLAINED 
{
  "meaning": "Requires a 10MHz SPI bus.",
  "purpose": "High-speed synchronous communication."
}
1.7

Score & ID

Generates keywords and assigns a permanent, unique tracking number to every rule.

What this stage is doing

  • Picks out 3 to 8 search keywords so the rule is easy to find later.
  • Gives a confidence score to show how sure the AI is that this is a real rule.
  • Assigns a permanent barcode-like ID number (e.g., REQ-001) to each rule.

Why it is useful

It makes the document searchable and allows teams to track the progress of a specific rule over time.

Problem Solved

The Lost Item Problem: Without ID numbers, rules get lost or confused. This guarantees every single requirement can be tracked from start to finish.

Data State SCORED & ID'D 
{
  "id": "REQ-HW-INT-042",
  "confidence": 0.95,
  "keywords": ["spi", "10mhz", "bus"]
}
1.8

Global Dedup

Finds rules that are repeated across the document and merges them into one single rule.

What this stage is doing

It compares every single rule against all the others. If it finds two rules that say the exact same thing, it deletes the copy and merges them together.

Why it is useful

It creates a clean, focused list where every instruction is defined exactly once, preventing double-work.

Problem Solved

The Copy-Paste Problem: Large documents often repeat the same instruction in five different chapters. This cleans up the mess and creates one single source of truth.

Data State READY 
{
  "id": "REQ-HW-INT-042",
  "status": "master",
  "merged": 1,
  "final_text": "The system shall support SPI, 10MHz."
}
